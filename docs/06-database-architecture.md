# 06 - Database Architecture

## 1. Database Design Goals

- Use PostgreSQL as the source of truth.
- Use UUID primary keys for distributed-safe identifiers.
- Link application users to Supabase Auth users through `auth_user_id`.
- Include `gym_id` on operational and content tables to support future multi-branch or SaaS expansion.
- Prefer soft delete/archive for business records with financial, attendance, or audit relevance.
- Use row-level security policies for role, ownership, and gym-scoped access.
- Use indexes on foreign keys, dates, statuses, and search fields.

## 2. Common Conventions

| Convention | Description |
| --- | --- |
| Primary keys | `id uuid primary key` generated by database. |
| Timestamps | `created_at timestamptz`, `updated_at timestamptz`, `deleted_at timestamptz nullable` where soft delete applies. |
| Status fields | Text constrained by check constraints or PostgreSQL enums. |
| Money | Store amount in smallest currency unit as integer, for example paise for INR. Also store currency code. |
| Metadata | Use `jsonb` only for provider metadata, flexible settings, or plan details that do not require frequent filtering. |
| Search | Use lower-cased generated/search columns or trigram indexes for names, phones, emails where needed. |
| Audit | Critical mutations should write to `audit_logs`. |

- `id`

Foreign keys:

## 14. ER Diagram Description

The entity model is centered on `gyms` as the tenant boundary. `users` belong to a gym except platform-level super admins, and `user_roles` grants role access. `members` and `trainers` are role-specific profiles connected to `users`.

Membership commerce flows from `membership_plans` to `memberships` and `payments`. A member purchases or is assigned a membership plan, creating a `memberships` row. A payment may be recorded online through Razorpay or offline by staff. Razorpay webhook events are stored in `payment_events`, and verified payment capture activates or renews the membership.

Operational engagement flows through `attendance`, `classes`, and `class_bookings`. Classes are assigned to trainers, members book classes, and attendance can reference either a general gym check-in or a class booking.

Fitness delivery flows through `workout_plans` and `diet_plans`, each linking a member and a trainer. Only assigned trainers, admins, and the owning member should see the appropriate plan data.

Marketing and CRM data flows through public content tables (`blogs`, `gallery`, `testimonials`) and the `leads` table. Leads may be converted into member records. Notifications connect user-facing events to in-app and email communication.

Audit logs sit beside all critical business processes and record who changed important data, which entity changed, and before/after state where appropriate.

## 15. Recommended Row-Level Security Policy Groups

| Table Group | Policy Intent |
| --- | --- |
| Public content | Guests can read published rows only. Admins can manage rows for their gym. |
| Users and roles | Users can read own profile. Admins can manage gym users. Super admins can manage all. |
| Members | Members can read/update limited own fields. Trainers can read assigned members. Staff can manage based on role. |
| Payments | Members can read own payments. Staff can read/manage according to role. Provider webhook service role can update payment status. |
| Attendance | Members can read own attendance. Staff can manage gym attendance. Trainers can access assigned class/member attendance. |
| Plans | Members can read own published plans. Trainers can manage assigned member plans. Admins can review. |
| Leads | Guests can insert only through controlled server action/route. Staff can manage gym leads. |
| Audit logs | Admin read only, super admin full read. No client insert except server-side audit service. |
